# 嵌入式系统 Skill 路由

默认使用中文回答和生成说明。仅在用户明确要求英文、项目文件原本使用英文注释，或对外交付规范要求英文时，才切换为英文。

将此 skill 作为嵌入式固件任务的一级入口。保持轻量：先识别任务领域，再加载本目录下最匹配的二级 skill。

需要更明确的分流表时，读取根目录 `readme.md`。不要为了一个任务重复加载多个同级 skill；只选择主 skill，并在确有必要时补充一个辅助 skill。

## 路由规则

- 使用 `c99-standard-c` 处理 C99 模块设计、`.c/.h` 生成、编码风格、内存安全、可移植性、文件编码、ISR/并发规则和底层 C 重构。
- 使用 `review` 处理代码评审、风险评估、检查清单式审查、文档/变更记录检查、MaJerle/c-code-style 风格参考和回归测试建议。
- 使用 `dsp` 处理滤波器、FFT、定点运算、数值稳定性、采样、量化、饱和和实时信号处理路径。
- 使用 `arm` 处理 ARM Cortex 启动、CMSIS、寄存器、NVIC、Fault、MPU/cache、链接/启动问题和厂商 HAL 边界。
- 使用 `ethercat` 处理 EtherCAT 从站固件、PDO/SDO/CoE/FoE、分布式时钟、ESI/对象字典、ESC/SPI/PDI 和状态机行为。
- 使用 `os` 处理 RTOS 任务、调度器、中断、队列、互斥锁、定时器、裸机 superloop 和优先级/延迟分析。
- 使用 `motorcontrol` 处理 FOC、PID 环、SVPWM、电流/速度/位置环、编码器/Hall/观测器、驱动保护和调试投产。

## 默认原则

- 默认采用保守嵌入式工程顺序：正确性、安全性、可测试性、可维护性、可移植性，然后才是经过测量的性能优化。
- 保持现有项目风格和已验证代码；只有存在缺陷、需求、安全风险、测试失败或用户明确要求时才修改。
- 源文件编码默认按 GBK 处理，除非项目证据表明并非如此。转换编码前必须询问用户。
- 生成 C 代码时优先使用 ISO C99、显式位宽整数、有界资源使用、静态所有权和清晰的硬件副作用边界。
- 做架构分析时明确 MCU、编译器、RTOS、总线、内存、实时预算和硬件依赖假设。

## 输出要求

- 实现类任务给出可落地文件或补丁，并补充验证步骤。
- 评审类任务先列问题，按严重度排序，有文件时给出文件/行号。
- 领域交叉任务要明确使用的组合 skill，例如“使用 `motorcontrol`、`c99-standard-c` 和 `review`”。
