# MaJerle C Code Style 中文评审参考

来源：[MaJerle/c-code-style](https://github.com/MaJerle/c-code-style)。此文件是面向 `review` skill 的中文化评审摘要，不是逐字翻译，也不是项目强制格式规范。

## 使用边界

- 只在用户要求代码风格评审、格式化规则、Doxygen、命名、头/源文件组织，或明确提到 MaJerle/c-code-style 时读取。
- 不替代 `c99-standard-c`。涉及语言基线、`stdbool.h`、动态内存、VLA、ISR、寄存器、并发、安全和嵌入式 C99 实现规则时，以 `c99-standard-c` 为准。
- MaJerle 原仓库偏向 C11 风格；本 embedded 技能体系仍以 C99 为默认。不得为了匹配 MaJerle 而强制切换语言标准。
- MaJerle 部分建议与当前 C99 skill 不一致，例如是否使用 `stdbool.h`、大括号位置、函数返回类型换行、指针星号靠近类型。评审时必须先看项目既有风格。
- 未采用该风格的项目中，风格差异通常最多作为 P3 建议；只有影响可读性、维护性、自动格式化、生成文档或误导行为时，才升级严重度。

## 最重要原则

- 修改已有代码时，先观察周围代码并模仿当前项目风格。
- 不因为个人偏好或外部风格指南，对稳定代码做全文件格式化、批量重命名、批量换行或缩进重排。
- 如果项目已配置 `.clang-format`、AStyle、Eclipse formatter 或 CI 格式检查，评审应优先依据项目配置。

## 格式与排版检查

- 缩进应统一，MaJerle 建议 4 空格且不用 tab；若项目已有不同配置，以项目配置为准。
- 关键字和左括号之间保留 1 个空格，例如 `if (condition)`、`for (...)`。
- 函数名和实参左括号之间不加空格，例如 `sum(4, 3)`。
- 比较、赋值和常见二元运算符两侧保留空格；逗号后保留空格。
- 复合语句必须使用花括号，即使只有一条语句。
- `switch` 必须有 `default`，每个 `case/default` 的缩进和 `break/return` 位置要一致。
- 空循环必须显式写成空花括号，并说明等待条件；不要用裸分号制造不明显的空循环。

## 命名与声明检查

- 变量、函数、结构体和 typedef 名称应使用小写字母和下划线，除非项目已有不同命名规范。
- 不使用以 `_` 或 `__` 开头的自定义标识符，因为这些名称可能被 C 实现或标准库保留。
- 宏名使用全大写和下划线。
- 枚举成员通常使用全大写。
- 函数指针 typedef 可使用 `_fn` 后缀；是否强制取决于项目规范。
- 公共函数必须在头文件中有声明；文件私有函数必须保持 `static`。
- 局部变量声明位置遵守项目规范。MaJerle 建议在 block 开头集中声明；现有 C99 skill 允许靠近首次使用。评审时以项目局部一致性为准。

## 类型、指针与常量检查

- 非 `char`、`float`、`double` 的固定宽度整数优先使用 `stdint.h` 类型。
- 长度和容量使用 `size_t`。
- 不修改指向数据的指针参数应加 `const`；同时不修改指针自身时可以使用 `const void* const` 等形式，但不要牺牲可读性。
- 泛型数据接口可使用 `void*` 或 `const void*`，实现内部再做受控转换。
- 指针必须与 `NULL` 显式比较；这条可作为风格建议，但不得掩盖真正的空指针路径问题。
- `sizeof` 使用括号，避免表达式歧义。
- 函数返回 `void*` 时，C 中不需要强制转换为目标指针类型；多余 cast 可能隐藏 include 或类型问题。

## 变量初始化与嵌入式启动风险

- MaJerle 强调不要依赖全局变量显式初始化值，尤其是分散 RAM、自定义 section 或复杂启动脚本场景。
- 评审时检查模块是否有明确 init 函数，并在 init 中设置运行期默认值。
- 对于 `.bss`、`.data`、自定义 RAM section、备份 SRAM、DMA buffer、noinit section，检查启动文件和链接脚本是否保证预期初始化。
- 静态常量、查表和放入 Flash/ROM 的数据不应被误判为普通全局运行状态。

## 注释与 Doxygen 检查

- MaJerle 建议统一使用 `/* ... */` 注释，避免 `//`。若项目允许 `//`，只作为风格一致性建议。
- 多行注释格式应统一，避免注释块缩进和星号风格混乱。
- 公共 API、结构体、枚举、宏和关键静态函数应具备有意义的 Doxygen 或项目约定注释。
- 函数文档至少说明用途、参数输入/输出方向、返回值和特殊约束。
- 注释必须解释意图、边界、硬件时序、同步、失败路径或安全原因，不重复翻译代码。
- 中文或英文注释语言以项目规范为准；本 embedded skill 默认中文输出，但不能擅自改变项目源码注释语言。

## 结构体、枚举与初始化检查

- 结构体成员命名和注释应统一。
- typedef 命名是否使用 `_t` 后缀按项目规范判断；不要和已有代码混用。
- 复杂结构体初始化建议使用 C99 designated initializer。
- 多行复杂初始化建议保留尾随逗号，便于格式化工具稳定处理。
- 跨通信、Flash、寄存器或 ABI 边界的结构体不得只依赖 C 布局；必须检查对齐、填充、大小端和版本兼容性。

## 宏与预处理检查

- 数字常量应使用具名宏、枚举或 `static const`，避免魔法数。
- 宏参数和最终表达式必须加括号。
- 多语句宏必须使用 `do { ... } while (0)` 包裹。
- 条件编译建议使用 `#if defined(X)` / `#if !defined(X)`，并在 `#else/#endif` 注明条件含义。
- 不要让预处理分支破坏缩进、可读性或静态分析覆盖。
- 能用 `static inline` 且需要类型检查时，优先考虑替代函数式宏；这与 `c99-standard-c` 保持一致。

## 头文件与源文件组织检查

- 头文件必须有 include guard。
- 头文件必须在 C++ 兼容场景下使用 `extern "C"` 包裹声明，且 include 放在 `extern "C"` 之外。
- 标准库头文件用 `<...>`，项目头文件用 `"..."`。
- 头文件只暴露公共类型、宏和函数，不放私有声明或实现细节。
- 头文件包含应足够自包含，但不要包含无关依赖。
- 源文件优先包含对应头文件，再包含其他项目头和标准头。
- 不允许在 `.c` 中包含另一个 `.c`。
- 文件末尾保留单个换行。

## 与 `c99-standard-c` 的兼容优先级

评审时按以下顺序处理冲突：

1. 项目已有规范、CI、格式化配置和周围代码风格。
2. `c99-standard-c` 的安全、实时性、C99、编码、并发、寄存器和可移植性规则。
3. 本文件的 MaJerle 风格建议。

典型冲突处理：

- MaJerle 建议 C11；本体系默认 C99，不能强制升级。
- MaJerle 不建议 `stdbool.h`；本 C99 skill 允许并推荐 `bool`，不能把 `bool` 当成问题。
- MaJerle 使用同行左花括号；本 C99 skill 允许项目统一选择，不把大括号位置作为硬性问题。
- MaJerle 有特定指针星号风格；本 C99 skill 允许项目统一选择，不因星号位置单独报高严重度。
- MaJerle 建议局部变量 block 开头声明；C99 可允许靠近首次使用，评审时看项目一致性和可读性。

## 输出建议

- 风格问题通常放在 P3，除非它会导致工具失败、审查困难、误解硬件行为、隐藏 bug 或破坏公共接口。
- 同一类格式问题不要逐行刷屏；合并成一个 finding，并给 2-3 个代表位置。
- 如果项目没有采用 MaJerle 风格，只给迁移建议和风险，不要求大规模修改。
- 如建议引入 `.clang-format`，必须说明这会造成大范围 diff，应单独提交，不夹带功能修改。
