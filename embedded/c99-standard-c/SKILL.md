# 嵌入式 C99 标准 C

默认使用中文回答和生成说明。仅当项目既有代码注释/接口要求英文，或用户明确要求英文时，才使用英文。

使用此 skill 处理嵌入式 C99 实现任务。原始迁移的完整嵌入式 C99 规范保存在 `references/legacy-embedded-c99-skill.md`；当任务需要详细规则、模板或检查清单时读取该文件。

## 核心流程

1. 识别上下文：新模块、遗留代码修改、bug 修复、驱动/BSP、ISR、协议解析、控制算法、RTOS 集成或安全路径。
2. 先读公共头文件，再看实现。梳理公共 API、私有状态、全局所有权、中断/共享数据访问、硬件副作用和测试切入点。
3. 保持局部风格和已验证代码。不要顺手重构、大范围改名、格式化扰动或编码转换。
4. 做满足需求的最小完整修改，并保证行为可验证。
5. 补充验证项：编译选项、告警、静态检查、单元测试、边界条件、ISR/并发检查和必要的 HIL 项。

## 强制规则

- 源文件、头文件和 Markdown 参考文件默认编码为 GBK。转换为 UTF-8 或其他编码前必须询问用户。
- 语言基线为 ISO C99。优先使用 `stdint.h`、`stdbool.h`、`stddef.h` 和显式位宽整数。
- 禁止在实时路径使用递归、VLA、默认动态分配、无界循环、ISR 阻塞调用或不可控资源消耗。
- 公共头文件必须自包含且只包含声明。不得在公共头文件定义变量或放入私有实现细节。
- 文件私有函数和状态必须使用 `static`。
- 公共接口要检查指针参数和范围约束。可恢复失败使用明确错误码返回。
- 所有外部输入默认不可信。解析前验证长度、版本、大小端、CRC/checksum 和边界。
- 硬件寄存器、DMA、cache、ISR 和共享内存访问必须显式表达副作用和同步边界。

## C99 模块默认做法

- 使用 include guard、C++ 兼容的 `extern "C"` 和最小 include。
- 函数保持短小、单一职责、低嵌套。将复杂状态机和硬件胶水与纯逻辑分离。
- 参数较多时使用配置结构体。
- 避免魔法数。集中管理寄存器掩码、时序常量、缩放系数和协议常量。
- 需要类型检查时优先使用 `static inline`，少用函数式宏。
- 注释解释意图、时序、硬件约束、安全原因和非显然算法，不逐行翻译代码。
